# Implement

Step 3 is a two-phase TDD process. The software-engineer owns both phases entirely.

- **Step 3A (Functional TDD)**: RED → GREEN → REFACTOR for design correctness. Quality Gate A checks design only. Hand off to SA for design review.
- **Step 3B (Completion)**: Coverage, lint, type checking, docstrings. Quality Gate B checks cosmetic tooling. Hand off to SA for completion verification.

This step is a subflow defined in `.flowr/flows/tdd-cycle.yaml` with states: **setup → red → green → refactor → design-review → coverage → completion**.

## When to Use

Load this skill when continuing Step 3 for an in-progress feature. Architecture stubs must already exist (created by the system-architect at Step 2).

## Two-Phase Quality Priority

See [[software-craft/code-quality#key-takeaways]] for the full priority order. Summary:

1. **Step 3A**: Design correctness (YAGNI > KISS > DRY > SOLID > OC > patterns) — verify with `test-fast` and feature-type run only.
2. **Step 3B**: Coverage threshold, lint, pyright, docstrings — verify after design approval only.

Never run lint, format, static-check, or coverage during Step 3A. Those are Step 3B concerns. Running them on code that might be redesigned is wasted effort.

## Step 3A — Functional TDD

### Prerequisites

- [ ] Exactly one `.feature` in `in-progress/`. If not present, load `skill select-feature`
- [ ] Architecture stubs present in `<package>/` (committed by Step 2)

**Required reads**:

| Read | Why |
|---|---|
| `docs/system.md` | Current system structure and constraints |
| In-progress `.feature` file | Acceptance criteria |
| Test stub files in `tests/features/<feature_slug>/` | Generated by pytest-beehive at Step 2; if missing, re-run `uv run task test-fast` |

ADR details are available on demand: reference Key Decisions in `system.md`, then read specific ADR files only when a decision needs deeper context.

### Setup

Create or switch to the feature branch:

1. `git branch --show-current` — if on `feat/<stem>` or `fix/<stem>`, proceed
2. If on `main` or wrong branch: `git checkout -b feat/<stem>` (new) or `git checkout feat/<stem>` (existing)
3. Verify architecture stubs exist: `ls <package>/`

### Build Test List

1. List all `@id` tags from in-progress `.feature` file
2. Order: fewest dependencies first; most impactful within that set
3. Each `@id` = one TODO item, status: `pending`
4. Confirm each `@id` has a corresponding skipped stub in `tests/features/<feature_slug>/`

### Inner Loop — One @id at a time

WIP limit: exactly one `in_progress` at all times.

For each pending `@id`, follow the TDD cycle. See [[software-craft/tdd]] for the full cycle rules, green bar rule, and two-hats discipline.

```
RED    → write failing test for this @id
GREEN  → write minimum code to pass (YAGNI + KISS only)
REFACTOR → design refactoring ONLY (SOLID, OC, DRY, KISS, YAGNI, patterns)
         → do NOT apply cosmetic fixes (import ordering, docstrings, type annotations, format)
```

After each `@id` completes REFACTOR: run `uv run task test-fast`. If more `@id`s remain, return to RED for the next one.

If a smell is detected during REFACTOR, load `skill refactor` and apply one catalogue entry at a time. If a pattern smell is detected, load `skill apply-patterns`.

If a spec gap is discovered during implementation: **do not extend criteria yourself** — escalate to PO. Document the gap in the handoff message.

### Quality Gate A (before design review)

```bash
uv run task test-fast
```

Feature-type verification (choose one):

| Feature Type | Command |
|---|---|
| CLI | `timeout 10s uv run task run` |
| Library | `uv run python -c "import <package>; <public_api_call>"` |
| Mixed | Both commands above |

All must pass before Design Self-Declaration.

### Design Self-Declaration

Communicate verbally to the system-architect. Cover design correctness only — YAGNI, KISS, DRY, SOLID, OC, patterns, and semantic alignment. See [[software-craft/self-declaration]] for the full 25-item checklist.

This declaration covers **design correctness only** — not lint, coverage, type checking, or docstrings. Those are declared in the Completion Declaration at Step 3B.

### Branch Hygiene (before handoff)

1. `git status` — working tree must be clean
2. `git branch --show-current` — must be `feat/<stem>` or `fix/<stem>`
3. `git log main..HEAD --oneline` — must show 1+ commits
4. `git push origin $(git branch --show-current)`

### Hand off to Step 4 (Design Verification)

Signal completion to the system-architect. Provide:
- Feature file path
- Design Self-Declaration (verbally)
- Summary of what was implemented
- Feature type (CLI / Library / Mixed)

If REJECTED → address failures and re-enter the TDD loop from RED. If APPROVED → proceed to Step 3B.

## Step 3B — Completion

Step 3B starts after the system-architect APPROVED the design at Step 4. Same branch (`feat/<stem>` or `fix/<stem>`).

### Coverage Gate

```bash
uv run task test-coverage
```

If coverage is below the threshold configured in `pyproject.toml`: add tests in `tests/unit/` for uncovered branches. Do NOT add `@id` tests for coverage — those are for acceptance criteria only. Re-run `test-coverage` until the threshold is met.

### Quality Gate B (before completion handoff)

```bash
uv run task lint
uv run task static-check
uv run task test-fast
```

Feature-type verification (same as Step 3A). All must pass before Completion Declaration.

### Completion Declaration

Communicate verbally to the system-architect. Answer for each item:

1. Coverage threshold met — `uv run task test-coverage` exits 0
2. Lint clean — `uv run task lint` exits 0 (no `noqa`, no `type: ignore`)
3. Type checking clean — `uv run task static-check` exits 0 (0 pyright errors)
4. Public API documented — all public classes and methods have docstrings

### Branch Hygiene (before completion handoff)

Same as Step 3A: clean working tree, correct branch, commits ahead, pushed to origin.

### Hand off to Step 4B (Completion Verification)

Signal completion to the system-architect. Provide:
- Feature file path
- Completion Declaration (4 items above)
- Summary of coverage additions and cosmetic fixes made

## Signature Design

Architecture stubs use `...` for bodies. GREEN phase replaces `...` with minimum implementation. REFACTOR phase cleans up the result.

Use Python Protocols for external dependencies — never depend on a concrete class directly. See [[architecture/domain-stubs]] for stub conventions and Protocol patterns.
