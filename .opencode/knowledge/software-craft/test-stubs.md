# Test Stubs

## Key Takeaways

- Test stubs are auto-generated by `beehave generate <feature_id>` from the feature file; no manual stub creation is needed.
- pytest-beehave uses title-based mapping: each Example title becomes a test function named `test_<scenario_title_slug>` (e.g., `test_VAT_is_applied_at_the_correct_rate`).
- Structural traceability (every Example has a test, no orphan tests) is verified by `beehave check`, not manually.

## Concepts

**Title-Based Mapping**. pytest-beehave maps each Example in the feature file to a test function by title. The function name is derived from the scenario title as a slug (e.g., Example: "VAT is applied at the correct rate" → `test_VAT_is_applied_at_the_correct_rate`). This replaces the previous `@id` tag system. Titles must be unique within a feature file.

**Auto-Generated Stubs**. When `beehave generate <feature_id>` runs, pytest-beehave reads the feature files and creates test stubs automatically. Each stub has an `...` (Ellipsis) body. During pytest collection, pytest-beehave auto-skips any test function with an `...` body — no `@pytest.mark.skip` decorator is needed. The SE replaces the `...` body with the test implementation during the RED phase.

**beehave check**. The `beehave check` command (from the `beehave` library, not to be confused with the `behave` BDD framework) verifies structural traceability. It enforces: unmapped-scenario, unmapped-test, misplaced-test, missing-placeholder, missing-literal, example-mismatch. The `stubs-traceable` condition on the `create-py-stubs.done` transition uses `beehave-check-passes: ==verified`.

**Test File Layout**. pytest-beehave organizes tests as: Feature title → directory, Rule → test file, Scenario → function name. Test files are placed in `tests/features/<feature_slug>/<rule_slug>_test.py`.
